# Warehouse Match Runs: Execution & Analysis Guide

This directory documents match runs, thresholds, and analysis for lead-to-POS matching across Costco warehouses.

## Quick Start

### Trigger a Match Run

**GitHub Actions UI**:
1. Actions → lead_match_semantic_workflow.yml → Run workflow
2. Set warehouse (e.g., "115") or leave blank for all
3. Wait for completion

**Cloud Workflow**: After semantic workflow triggers lead_match_workflow

### Generate Analysis & Reports

After match run completes:

1. **Automatic** (if triggered via workflow_run)
   - lead_match_analysis.yml runs automatically
   - Generates business report + Gemini analysis

2. **Manual**
   - Actions → lead_match_analysis.yml → Run workflow
   - Set warehouse and run_id
   - Select generate_report and generate_analysis flags

### View Results

- **Business report**: `gs://lead-match-ctoteam/reports/lead_match/ctoteam/<wh>/<run>/report.md`
- **Matches CSV**: `gs://lead-match-ctoteam/reports/lead_match/ctoteam/<wh>/<run>/matches.csv`
- **Gemini analysis**: `gs://lead-match-ctoteam/reports/lead_match/ctoteam/<wh>/<run>/comparative_analysis.md`
- **Reporting app**: `https://<service>/analysis?warehouse=<wh>&run_id=<run>`

## Workflow Architecture: Separated Concerns

### 1. Lead Match Semantic Workflow (GitHub Actions)

**Role**: Build runtime + deploy + trigger matching execution

**Inputs**: warehouse, deploy_runtime, deploy_workflow, prepare_readiness, run_limits

**Responsibilities**:
- Build runtime image in Cloud Build (optional)
- Update Cloud Run matching jobs (optional)
- Deploy Cloud Workflow YAML (optional)
- Create GCS preflight readiness contract (optional)
- Trigger lead_match_workflow Cloud Workflow
- Poll until completion
- Report execution ID

**Outputs**:
- Execution ID
- match_run_id (github-<run>-<attempt>-<wh>)
- _READY.json in GCS (signals to analysis workflow)

### 2. Lead Match Workflow (Cloud Workflow - Google Workflows)

**Role**: Parallel matching execution (exact + embeddings + fuzzy)

**No longer includes**: Report generation (moved to analysis workflow)

**Jobs**:
1. Exact match → Cloud Run job
2. Lead embeddings → Cloud Run job (parallel)
3. POS embeddings → Cloud Run job (parallel)
4. Ensure indexes → pgvector HNSW
5. Fuzzy match → Cloud Run job

**Outputs**:
- Cloud SQL: updates to lead, pos_embeddings, leads_embeddings, transaction
- GCS artifacts: summary.json, matches.csv, _READY.json
- No longer writes: report.md, analysis artifacts

### 3. Lead Match Analysis Workflow (GitHub Actions)

**Role**: Post-hoc reporting & analysis (decoupled from execution)

**Inputs**: warehouse, run_id, generate_report, generate_analysis

**Responsibilities**:
- Verify _READY.json (match run complete, safe to analyze)
- Generate business report (if enabled)
  - Calls: `scripts/generate_fuzzy_match_business_report.py`
  - Outputs: report.md, fuzzy_match_business_report.md, summary.json
- Generate Gemini analysis (if enabled)
  - Calls: `scripts/analyze_match_distribution.py`
  - Deterministic: per-row reasoning, distribution facts
  - Gemini 3.5 Flash: narrative (4 post-id signals + actions)
  - Outputs: comparative_analysis.md
- Upload to GCS
- Report results

**Triggers**:
- Manual dispatch (GitHub UI)
- Automatic after semantic workflow succeeds (workflow_run event)

## Why Split?

**Before**: Semantic Workflow → Cloud Workflow → Cloud Run Report Job (blocking, coupled)

**After**: 
- Semantic Workflow → Cloud Workflow (fast, clean execution)
- ↓
- Lead Match Analysis (post-hoc, flexible, independent)

**Benefits**:
- ✅ Match execution fast (no report overhead)
- ✅ Analysis can be re-run without matching
- ✅ Business report + Gemini analysis independently toggleable
- ✅ No race condition (gated by _READY.json)
- ✅ Lighter matching pipeline (exact + embeddings + fuzzy only)

## Thresholds & Business Rules

**Source**: `lead_match_runtime/lead_to_pos_match_rules.json`

| Band | Score Range | Lifecycle | Meaning |
|------|-------------|-----------|---------|
| Exact | 100 | Closed - Match | Deterministic, authoritative |
| Matching High | 90-99.999 | Closed - Match | Strong AI inference |
| Potential (Medium) | 85-89.999 | Potential | Likely, needs review |
| Potential (Low) | 70-84.999 | Potential | Possible, needs review |
| No Match | 0-69.999 | No Match | Insufficient evidence |

**Scoring Formula**:
```
final_score = (4 * full_address_score + 3 * business_name_score) / 7
```

- Weights from rules JSON (addr 4, name 3, denom 7)
- Recall gate: combined_field >= 65 (pre-filter)
- **Never hardcoded** — JSON is source of truth

## Post-Identification Signals (from Gemini Analysis)

1. **Threshold Sensitivity** — Peak within 2 points of cutoff?
   - Action: Calibrate threshold with labeled set if yes

2. **Tail/Edge Quality** — How many weak matches (70-84.999)?
   - Action: Validate recall gate and data quality

3. **Artifacts** — Spikes or gaps in histogram?
   - Action: Debug if found; else confirm no artifact

4. **Review Workload** — Potential + Manual Review row count?
   - Action: Size ServiceNow queue, fleet capacity planning
